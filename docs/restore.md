# restore

Restore guide for a freshly installed system using the custom Kinoite image.

The image already includes `restic`, `resticprofile`, and `task`, so no manual tool installation is required.

## Overview

Restore recovers the full `$HOME` from a restic snapshot and then automatically re-applies saved workstation state:

- layered `rpm-ostree` packages
- Flatpak apps
- VS Code extensions

The `resticprofile` restore config (`[default.restore]` in `restic/profiles.toml`) uses:
- `target = "/"` with absolute path mapping
- `delete = true` - files not present in the snapshot are removed
- `exclude = ["$HOME/.config/restic"]` - local credentials are never overwritten
- `run-after = ".../restore/bootstrap.sh"` - workstation state is re-applied automatically after restore

## ⚠️ Prerequisites

**Restore is impossible without this information:**

| Item | Where to keep it |
| --- | --- |
| `~/.config/restic/password` | USB drive, password manager, or another secure offline medium |
| S3 credentials (repository URL, access key, secret) | Same secure source |

These files are excluded from backup and intentionally not overwritten during restore.

## Safety Note

Run restore from a TTY (`Ctrl`+`Alt`+`F3`) or **before first graphical login**,
so running desktop processes do not overwrite files while they are being restored.

## Quickstart

```bash
# 1. Clone repo
git clone https://github.com/mwmahlberg/backup.git ~/.local/share/backup
cd ~/.local/share/backup

# 2. Create config (credentials from USB/password manager)
RESTIC_REPOSITORY=s3:fra1.digitaloceanspaces.com/mwmbackups \
  AWS_ACCESS_KEY_ID=your-access-key \
  AWS_SECRET_ACCESS_KEY=your-secret-key \
  RESTIC_PASSWORD=your-restic-password \
  task restore:init

# 3. Run restore
task restore:run

# 4. Reboot
sudo systemctl reboot

# 5. After reboot: re-enable backup schedules
task restore:schedule
```

## Step by Step

### 1) Create configuration

```bash
git clone https://github.com/mwmahlberg/backup.git ~/.local/share/backup
cd ~/.local/share/backup
```

```bash
RESTIC_REPOSITORY=s3:fra1.digitaloceanspaces.com/mwmbackups \
  AWS_ACCESS_KEY_ID=your-access-key \
  AWS_SECRET_ACCESS_KEY=your-secret-key \
  RESTIC_PASSWORD=your-restic-password \
  task restore:init
```

`restore:init` creates:
- `~/.config/restic/password` (chmod 600)
- `~/.config/restic/env` (chmod 600)
- `~/.config/resticprofile/profiles.toml` (symlink to `restic/profiles.toml`)

If the repository already exists, `restore:init` runs `git pull` instead.

### 2) Check snapshots (optional)

```bash
set -a; source ~/.config/restic/env; set +a
restic \
  -r "$RESTIC_REPOSITORY" \
  --password-file ~/.config/restic/password \
  snapshots
```

### 3) Run restore

Latest snapshot:

```bash
task restore:run
```

Specific snapshot ID (manual via `resticprofile`):

```bash
resticprofile -c ~/.local/share/backup/restic/profiles.toml restore 70e69674
```

`restore/bootstrap.sh` runs automatically after a successful restore and reapplies layered packages, Flatpaks, and VS Code extensions.

### 4) Update repository (if restored from older snapshot)

```bash
git -C ~/.local/share/backup pull --ff-only
```

### 5) Reboot and verify

```bash
sudo systemctl reboot
```

After reboot, verify shell config, dotfiles, project directories, and applications.

### 6) Re-enable backup schedules

```bash
task restore:schedule
```

Optional, to run timers without an active login session:

```bash
loginctl enable-linger "$USER"
```

## Re-apply Workstation State Manually

If needed, re-run bootstrap manually (for example after partial package failures):

```bash
task restore:bootstrap
```

## State Files

These files are generated by hooks in `restic/hooks/` during backup and stored in snapshots:

| File | Purpose |
| --- | --- |
| `~/.local/state/backup/layered-packages.txt` | layered rpm-ostree packages |
| `~/.local/state/backup/flatpaks.txt` | installed Flatpak apps |
| `~/.local/state/backup/vscode-extensions.txt` | VS Code extensions |
